## 解决度判据(每个意图:什么算"解决",不依赖深度业务知识)

> 这些判据是给 AI 的「可观察标准」。AI 据此判,评测者据此核 AI 判得对不对。

| 意图 | 算「解决」(yes) | 算「未解决」(no/partial) |
|------|------------------|--------------------------|
| 资产查询 | 给出了具体数值(利率/资产额),且用户未重问 | 只说"请查看XX页"未给值、用户下一轮重问 |
| 查持仓 | 答到"在哪看/怎么看"的具体路径 | 答非所问、用户继续追问位置 |
| 交易 | 给出可操作步骤(在哪点、怎么下单) | 只给概念不给步骤、用户问"那具体怎么操作" |
| 查开户营业部 | 给出查询/变更的具体路径或入口 | 含糊、用户重问 |
| 到价提醒 | 给出设置入口/操作步骤 | 只说"支持该功能"不给路径 |
| 问诊股 / 资金流向分析 | 给出明确口径/数据/解释,用户未追问"看不懂" | 用户下一轮"还是没懂""举个例子"→ partial |
| 自选 / 资金流向选股 | 给出操作路径或筛选结果 | 答非所问 |
| 咨询客服 | 给出转人工入口或客服电话 | 未给联系方式 |
| 拒识 | —(拒识类问题不应被证券业务承接,answer_resolved 通常 no) | 被错误地用证券答案回应 = 分发错 |
| 活动 | 给出权益内容 + 领取/使用入口 | 只列权益不给入口 |

## BU 承接判据(该不该证券承接 → should_dispatch_to_bu)

判断「这个问题该不该由证券 BU 承接」,只看当前问题 + context(前文),不看下一轮。

**该证券承接(should_dispatch_to_bu = true)**:与证券业务相关——
交易、持仓、资产/融资、营业部、行情/选股/问诊股、自选、到价提醒、证券活动权益、证券相关的人工客服等(即上面意图清单里除"拒识"外的所有)。

**该拒识(should_dispatch_to_bu = false)**:与证券业务无关——
- 明显他业务:手机充值、话费、银行卡变更/挂失、寿险/保单、信用卡账单等。
- 完全无关:闲聊、辱骂、测试性输入。

判据要写进 `dispatch_reason`,例:"问题是银行卡挂失,属银行业务,证券不应承接 → 该拒识"。

> 评测时:LLM 给的 `should_dispatch_to_bu` 会和日志事实 `dispatched_to_bu`(分发BU是否=证券)比对。
> 两类错误重点看:① 该证券承接却被拒识了(漏);② 该拒识却被证券承接了(误收,如"手机充值"被分进证券)。

## 答案 schema 备注

证券答案常见两类:活动权益卡(catalogId=mx-activity-result-multiple)、文本/FAQ/转人工(msgType=aat_text,双层嵌套+HTML)。已由 `scripts/answer_parser.py` 处理,无需手工解析。行情/选股数据卡若出现未知结构,降级为纯文本,判不准则转人工。
