# 证券 BU 评判细则

> 本文件 = 证券的「评判标准事实源」(skill 路线)。调整证券评测口径只改这里 + 业务分类 JSON。
> 四条铁律见 SKILL.md,本文件只补充证券特有的可观察判据。

## 业务分类清单(单一来源)

★ **业务分类(名称 + 定义)统一维护在 `categories/securities.json`**,本文件不再重复。
评测时读该 JSON 的 `categories`,据每个分类的 `definition` 识别客户问题属于哪个业务分类。
**调整业务分类(增删改名/改定义)只改这一个 JSON 文件**,主项目与 skill 两条路线都生效(主项目读 `backend/app/core/bu/data/securities.json`,内容与此处 JSON 一致)。

## 解决度判据(每个业务分类:什么算"解决",不依赖深度业务知识)

> 这些判据是给 AI 的「可观察标准」。AI 据此判,评测者据此核 AI 判得对不对。

| 业务分类 | 算「解决」(yes) | 算「未解决」(no/partial) |
|----------|------------------|--------------------------|
| 自选 | 给出自选股列表/表现数据 | 答非所问、用户重问 |
| 资产查询 | 给出具体数值(总资产/可用资金/资产额),且用户未重问 | 只说"请查看XX页"未给值、用户下一轮重问 |
| 交易 | 给出可操作步骤(在哪点、怎么下单/撤单) | 只给概念不给步骤、用户问"那具体怎么操作" |
| 查持仓 | 给出持仓/委托/成交的具体信息或查看路径 | 答非所问、用户继续追问 |
| 到价提醒 | 给出设置入口/操作步骤 | 只说"支持该功能"不给路径 |
| 股票主力资金流向分析 | 给出该股主力资金流入流出的明确数据/口径 | 用户下一轮"还是没懂""举个例子"→ partial |
| 主力资金流向选股 | 给出符合资金条件的选股结果 | 答非所问、无结果 |
| 查开户营业部 | 给出营业部信息或查询/变更入口 | 含糊、用户重问 |
| 问诊股 | 给出分析/诊断/公告/研报/资讯的明确内容,用户未追问"看不懂" | 答非所问、用户下一轮追问 → partial |
| 咨询客服 | 给出办理路径/转人工入口/规则说明 | 未给有效信息、用户重问 |

## BU 承接判据(该不该证券承接 → should_dispatch_to_bu)

判断「这个问题该不该由证券 BU 承接」,只看当前问题 + context(前文),不看下一轮。

**该证券承接(should_dispatch_to_bu = true)**:与证券业务相关——
交易、持仓、资产/融资、营业部、行情/选股/问诊股、自选、到价提醒、证券活动权益、证券相关的人工客服等(即上面意图清单里除"拒识"外的所有)。

**该拒识(should_dispatch_to_bu = false)**:与证券业务无关——
- 明显他业务:手机充值、话费、银行卡变更/挂失、寿险/保单、信用卡账单等。
- 完全无关:闲聊、辱骂、测试性输入。

判据要写进 `dispatch_reason`,例:"问题是银行卡挂失,属银行业务,证券不应承接 → 该拒识"。

> 评测时:LLM 给的 `should_dispatch_to_bu` 会和日志事实 `dispatched_to_bu`(分发BU是否=证券)比对。
> 两类错误重点看:① 该证券承接却被拒识了(漏);② 该拒识却被证券承接了(误收,如"手机充值"被分进证券)。

## 答案 schema 备注

答案**不做代码解析**(格式无法穷举:活动权益卡 / 文本FAQ / 转人工 / 行情数据卡 / 多层嵌套 / list 套 list…)。答案原文(JSON+标签)直接进样本的 `answer_text`,由 LLM 评测时自行读懂对用户可见的语义,忽略标签和无关字段。读不懂的判不准就转人工。
