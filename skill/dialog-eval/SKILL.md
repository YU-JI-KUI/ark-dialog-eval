# 对话评测 Skill

把运营手动给对话日志打标的工作做成自动评测。**本 skill 的核心不是"让 AI 聪明地判断",而是"让 AI 的判断可被信任"——即使评测的人不懂具体业务。**

## 0. 给评测者的话(为什么你能信任这个结果)

你可能不懂证券/寿险业务,无法直接判断「AI 说这个答案解决了用户问题」对不对。这没关系。本方法论的设计前提就是:**不依赖评测者的业务知识,也能产出可信结果。** 靠的是下面四条铁律——只要 AI 严格遵守,你看它的「判断依据」就能验证它有没有乱判,而不需要你自己懂业务。

---

## 1. 四条铁律(可信度的根基,任何 BU 都不可违背)

### 铁律一:每个判断必须附「依据」,禁止凭空下结论
AI 输出的每一个 yes/no/对/错,都必须同时给出一句**基于对话本身可观察到的事实**的依据。

- ❌ 禁止:`answer_resolved: yes`(无法验证,不可信)
- ✅ 要求:`answer_resolved: yes`,依据:「用户问融资利率,答案给出了具体数字6.5%,且下一轮用户未重问、未转人工」

**为什么**:评测者不懂业务,但能判断"依据本身合不合逻辑"。依据是把 AI 的黑箱判断变成可审计的关键。没有依据的判断一律视为无效。

### 铁律二:「是否解决」建立在可观察的代理信号上,不靠业务对错
判断答案是否解决用户,**不要求** AI 懂业务、不要求 AI 知道答案在事实上对不对。只看这些**任何人都能观察**的信号:

1. **相关性**:答案是否回应了用户的问题(答非所问 = 没解决)。
2. **完整性**:答案是否把话说完整(戛然而止、只给半句 = 没完整)。
3. **下游轨迹(最硬的信号)**:用户下一轮的行为(样本的 `next_user_turn` 字段)——
   - 道谢 / 不再追问 / 会话自然结束 → 倾向「解决」
   - 重问同一问题 / 表达不满 / 要求转人工 / 换种问法再问 → 倾向「未解决」
   - 这个信号**完全不需要懂业务**,且最接近用户真实感受。
   - ⚠️ `next_user_turn` **只能用于本维度(是否解决)**,严禁用于判意图/分发 —— 见铁律六。

`answer_resolved` 取值:`yes`(解决)/ `partial`(部分)/ `no`(没解决)/ `unknown`(信息不足无法判)。

### 铁律三:事实正确性一律「不可验证」,绝不假装懂业务
没有知识库,AI **不许**判断答案里的业务事实对不对(如"可转债要20个交易日日均10万"这个数字准不准)。

- prompt 里根本不让模型判业务事实,只判相关性 / 完整性 / 是否解决。
- 若答案的可信度严重依赖某个事实是否正确,标 `needs_human_review: true`,转人工。

**为什么**:AI 假装懂业务去判事实,是评测不可信的头号来源。宁可说"我判不了",也不许瞎判。

### 铁律四:不确定就转人工,不硬判
意图置信低 / 需要核实业务事实 / 疑似合规风险 / AI 自己拿不准 → `needs_human_review: true` 并说明原因。
目标是"只让人看难的那部分",而不是追求 100% 自动化。**该转人工却硬判,比转人工更伤可信度。**

### 铁律五:多轮对话必须结合上下文判意图,禁止孤立看单轮
多轮里,当前这轮的问题常常**指代上一轮 AI 的回答**,孤立看必判错。

- 例:第1轮用户「我的持仓」→ AI 返回持仓列表(1.平安银行 2.贵州茅台 …);第2轮用户「详细说一下**第二个**的走势」。
- 孤立看第2轮「第二个的走势」→ AI 会判成「其他」或瞎猜;**必须看 context 里上一轮 AI 答的列表**,才知道"第二个"=贵州茅台 → 这轮意图是「问诊股/查走势」,应分发到行情类 agent。

样本的 `context` 字段已包含前文每一轮的 `{turn, user(用户问), ai(AI答原文,不解析)}`。**判分发前,先读 context**:
- 当前问题含指代词(第二个/那个/它/换一个/上面那个/再看看)→ 必须回看 context 的 ai 答案确定指代对象。
- 当前问题是对上一轮的追问/筛选/细化 → 意图通常延续或细化上一轮,不要当成新的孤立问题。
- 判断依据(`dispatch_reason`)要写明"结合上一轮 AI 返回的 X,本轮指 Y"。
- 若 context 缺失或不足以确定指代 → `needs_human_review: true`。

### 铁律六:守住时间边界,禁止"数据穿越"
**被评测的 agent(线上当时)和做评测的你(事后复盘)信息边界不同,不要混。**

- 被评测的 agent 在回答第 N 轮时,**不可能知道**用户第 N+1 轮会说什么。
- 你在事后评测,**能看到**整通对话(包括 next_user_turn)。这是评测者的合法视角,不是作弊。

但要分清"哪个信息能用在哪个判断"——**判断的输入必须和 agent 当时的处境对齐**:

| 判断维度 | 允许看的信息 | 禁止看的信息 |
|----------|--------------|--------------|
| 意图 / 分发(③①) | 当前轮 + `context`(前文) | ❌ `next_user_turn`(未来) |
| 答案解决度(②) | 当前轮 + 答案 + **`next_user_turn`(用户事后反馈)** | — |

- 判**意图/分发**:agent 当时只有"当前问题 + 前文",所以你也只能用这些。**绝不能用 next_user_turn 反推意图**(那是用未来信息,agent 当时没有,判出来不公平也不真实)。
- 判**是否解决**:这本就是"事后看用户满不满意",`next_user_turn` 是用户的真实投票,必须用、也只在这里用。

一句话:**未来信息(下一轮)只能用来评"上一轮的效果",不能用来帮"上一轮做决策"。**

## 2. 评测三个维度(对每条对话)—— 一个漏斗

```
全部样本 N 条
  │
  ├─ 维度① BU 分发评测(对全部 N 条)
  │   LLM 判:这个问题(结合 context)该不该由【本 BU(如证券)】承接?
  │     · 该本 BU 承接 → 期望日志「分发BU=本BU」
  │     · 不该承接(与本 BU 无关) → 期望日志「分发BU≠本BU」即拒识
  │   比对日志事实(样本 dispatched_to_bu)→ 分发对/错 → 算 BU 分发准确率
  │   两类错误:① 该承接却拒识了(漏)  ② 该拒识却承接了(误收)
  │
  └─ 维度② 解决度评测(★只对「分发BU=本BU」的子集 —— 漏斗)
      · 维度③ 给问题打业务类型标签(只为切片,不判分类对错,无真值)
      · 判答案是否解决用户,没解决给原因
      → 每个业务类型的解决率 + 整体端到端解决率
```

| 维度 | 判什么 | 评测范围 | 依赖业务知识? |
|------|--------|----------|---------------|
| ① BU 分发 | 该不该由本 BU 承接(承接/拒识),与日志分发BU比对 | **全部样本** | 轻度(靠 rubric 的承接判据) |
| ② 答案解决度 | 答案是否解决用户,没解决给原因 | **仅分发BU=本BU的子集** | 不依赖(靠铁律二代理信号) |
| ③ 业务分类 | 打一个业务类型标签 | 仅子集 | 轻度;**只切片,不判对错(无真值)** |

**口径锁定(不可漂移)**:
- 「拒识」= 日志 `分发BU` **不是本 BU**(空或其他),不是某个意图类别。
- 业务类型只打标签做切片统计,**不评分类准不准**(Excel 无每条的业务类型真值)。
- 解决率分母 = **只有「分发BU=本BU」的问题**;拒识/分错的不进分母(端到端漏斗口径)。

各 BU 的「什么该承接 / 什么该拒识」承接判据 + 业务类型清单见 `reference/`(证券:`securities_rubric.md`,寿险:`life_rubric.md`)。

## 3. 执行流程

1. **读 BU**:确认评测哪个 BU(证券/寿险),加载两份:
   - `categories/<bu>.json` —— 业务分类清单(名称+定义),据此识别每个客户问题的业务分类。**这是业务分类的唯一来源,改分类只改这个 JSON。**
   - `reference/<bu>_rubric.md` —— 该 BU 的承接判据、解决度判据等评判细则。
2. **预处理(用脚本)**:
   - 调 `scripts/prepare.py 日志.xlsx` ——做会话重组(按会话ID分组、按轮次排序,拼前文上下文、记录下一轮),输出样本 JSON。**不做样本过滤**:上传什么评什么,按行删会破坏多轮上下文。
   - **答案不做代码解析**:答案格式无法穷举(JSON 渲染卡 / 多层嵌套 / 混入 HTML 标签 / 甚至 list 套 list),硬解必崩。脚本把答案列**原文**原样放进样本的 `answer_text`,**由你(LLM)在评测时自行读懂其中对用户可见的语义,忽略标签和无关字段**。前文 AI 答(context 里的 `ai`)同理是原文。
   - 脚本依赖 `pandas`、`openpyxl`(prepare)与 `scikit-learn`(metrics);内网若缺,走私服安装。
3. **判定模式**:数据里若有「分发是否正确/答案是否解决客户问题」等人工金标列且有「是/否」值 → 校准模式;否则 → 生产模式。
4. **逐条评测**:按三个维度 + 四条铁律输出判断(每条都带依据)。
5. **校准模式额外**:把 AI 判断与人工金标对齐,调 `scripts/metrics.py` 算一致率(准/召/F1 + Cohen's κ + 混淆矩阵),导出不一致 case。
6. **聚合洞察**:按业务分类切片统计解决率、分发准确率、需复核率。
7. **出报告**:严格按 `reference/report_template.md` 的结构与口径生成。

## 4. 单条评测的输出契约

每条对话输出如下 JSON(**每个判断都要有对应的 *_reason 依据**):

```json
{
  "should_dispatch_to_bu": true,
  "dispatch_reason": "★该不该本BU承接的依据(基于问题+context);该承接→true,该拒识→false",
  "business_type": "业务类型标签(仅切片用,见 rubric 清单;拒识类可填'非本BU')",
  "business_type_reason": "为什么打这个业务类型标签",
  "answer_relevant": true,
  "answer_complete": true,
  "answer_resolved": "yes|partial|no|unknown",
  "resolved_reason": "★依据:基于相关性/完整性/下游轨迹,不靠业务对错",
  "unresolved_cause": "没解决时的原因归类(答非所问/信息不全/事实存疑/分发错误等),解决了填空串",
  "needs_human_review": false,
  "review_reason": "需转人工的原因或空串"
}
```

字段说明:
- `should_dispatch_to_bu`:LLM 判「这个问题**该不该**由本 BU 承接」。该承接=true,该拒识=false。**只看当前问题+context,不看 next_user_turn(铁律六)**。
- **分发对不对由代码算**,不是 LLM 给:把 `should_dispatch_to_bu` 与日志事实 `dispatched_to_bu` 比对——一致=分发正确。两类错误:LLM 说该承接(true)但日志拒识了=漏;LLM 说该拒识(false)但日志承接了=误收。
- `answer_resolved` / `resolved_reason`:**只对「日志 dispatched_to_bu=true」的样本判**(漏斗);拒识/分错的样本此字段填 `unknown` 并在 resolved_reason 注明"非本BU承接,不评解决度"。
- `resolved_reason` 是可信度命门,必须显式引用代理信号(尤其用户下一轮行为),不许空泛。

## 5. 校准:让"可信"有数字(给"大家认可"的证据)

校准模式下,把 AI 判断和人工金标(运营标注的真值)做一致性统计:
- **Cohen's κ**:衡量 AI 与人工标注者的一致性,扣除了碰巧一致。**κ ≥ 0.6 视为可信,≥ 0.8 为高度一致。**
- 准确率 / 各类别 F1 / 混淆矩阵。
- **导出不一致 case 双向看**:既可能 AI 判错(改本 skill 的规则/细则),也可能人工标得糙(人工一致性本身也需确认)。别盲目追求 100% 一致。

κ 数字 + 不一致 case 列表,就是你向团队证明"这套评测可信"的客观材料——不需要你自己懂业务。

---

## 6. 沉淀方法论的纪律(本 skill 如何持续可信)

- 评判标准的任何调整,都改在 `reference/*_rubric.md` 和本文件,**不要散落在临时对话里**——这里是评判标准的唯一事实源。
- 每轮校准发现的系统性偏差(如"寿险理赔类 AI 总判过松"),回写进对应 rubric,下次生效。
- 新增 BU:复制一份 `reference/<新bu>_rubric.md`,填该 BU 的业务分类清单(JSON)与评判细则即可,四条铁律不变。
